# 4G 压力报警联动控制系统 Web 端开发计划

## 一、文档说明

- 项目名称：4G 压力报警联动控制系统
- 文档用途：明确 Web 管理后台首期开发范围、实现方式、联调边界与验收标准
- 依据文档：
  - docs/4G 压力报警模块 · 软件需求沟通文档.md
  - docs/4G 压力报警模块系统技术栈选择.md
  - docs/后端开发进度文档.md
- 当前更新时间：2026-04-03

## 二、当前现状

截至 2026-04-03，项目整体状态如下：

- backend/ 已完成第一版核心业务 API，可支撑 Web 管理后台联调
- docs/ 已有需求文档、技术栈文档、后端依赖文档与后端开发进度文档
- web/ 目录当前为空，尚未初始化前端工程
- 后端接口前缀为 /api/v1
- 当前认证方式为 JWT Bearer Token
- 当前角色模型包含：super_admin、manager、device_user
- 当前实际权限分层以 super_admin 与非 super_admin 为主，manager 可视为当前首期普通业务账号

结论：

- Web 端已具备开始开发和联调的条件
- 首期开发应以“尽快形成可登录、可查看、可控制的后台闭环”为目标
- 首期不建议一开始铺开所有后台管理与运维页面，应优先完成核心业务页面

## 三、首期开发目标

Web 端第一阶段按“核心 MVP”推进，目标如下：

1. 完成后台前端基础工程初始化
2. 完成登录鉴权与路由守卫
3. 完成首页仪表盘
4. 完成报警记录查询页
5. 完成设备列表与设备详情页
6. 完成模块继电器手动控制
7. 完成个人修改密码页
8. 与现有后端 /api/v1 接口打通首轮联调

首期不纳入范围：

- 用户管理
- 设备分组管理
- 设备归属分配
- 日志中心
- 调度任务页
- MQTT 状态页
- 导出页
- 复杂运维页
- 国际化
- 深色模式
- WebSocket 实时通信

## 四、技术栈与工程方案

根据现有文档，Web 端采用以下技术栈：

- Vue 3
- Vite
- TypeScript
- Vue Router
- Pinia
- Element Plus
- Axios
- ECharts

建议前端工程结构如下：

```
web/ ├─ src/ │  ├─ api/            # 接口封装与类型定义 │  ├─ assets/         # 样式、图标、静态资源 │  ├─ components/     # 通用组件 │  ├─ layouts/        # 后台布局 │  ├─ router/         # 路由与守卫 │  ├─ stores/         # Pinia 状态 │  ├─ utils/          # 工具函数 │  ├─ views/          # 页面 │  ├─ App.vue │  └─ main.ts ├─ public/ ├─ package.json ├─ tsconfig.json └─ vite.config.ts 
```

约定：

- 所有接口请求统一由 src/api/ 管理
- 所有认证状态统一由 Pinia 管理
- 所有受保护页面统一使用路由守卫控制访问
- 所有时间展示统一在前端格式化，不改变后端返回结构
- 所有页面统一具备加载态、空态、错误态

## 五、页面与接口规划

### 5.1 登录页

目标：

- 提供用户名、密码登录
- 登录成功后写入 token 并拉取当前用户信息
- 失败时展示明确错误提示

接口：

- POST /api/v1/auth/login
- GET /api/v1/auth/me

说明：

- 登录接口使用表单方式提交，不使用 JSON
- 登录成功后进入仪表盘页
- token 失效时自动回到登录页

### 5.2 仪表盘

目标：

- 展示系统概览、统计信息、设备运行状态、近期报警、近期指令
- 形成进入系统后的默认首页

接口：

- GET /api/v1/dashboard/home
- GET /api/v1/dashboard/charts
- GET /api/v1/dashboard/recent-alarms
- GET /api/v1/dashboard/recent-commands

展示内容：

- 设备总数
- 模块总数
- 在线模块数
- 离线模块数
- 当前告警数
- 待处理指令数
- 告警分布图
- 指令状态分布图
- 设备状态分布图
- 近期报警列表
- 近期指令列表

### 5.3 报警记录页

目标：

- 查看历史报警记录
- 支持分页与筛选
- 为运维和业务排查提供入口

接口：

- GET /api/v1/dashboard/alarms/page

展示字段：

- 设备名称
- 模块编号
- 报警类型
- 报警状态
- 联动状态
- 来源
- 消息内容
- 触发时间

### 5.4 设备列表页

目标：

- 展示当前用户可见设备
- 查看设备基础状态与模块数量
- 支持跳转设备详情

接口：

- GET /api/v1/devices
- GET /api/v1/devices/monitoring

展示字段：

- 设备名称
- SN
- 模块数量
- 在线模块数
- 离线模块数
- 最近报警类型
- 最近报警时间
- 当前设备状态

### 5.5 设备详情页

目标：

- 展示单设备下模块的实时状态
- 展示最近报警与最近指令
- 提供继电器手动控制入口

接口：

- GET /api/v1/devices/{device_id}
- GET /api/v1/dashboard/devices/{device_id}
- POST /api/v1/relay-commands

展示字段：

- 模块编号
- 在线状态
- 继电器状态
- 电量
- 电压
- 最后在线时间
- 最近报警记录
- 最近控制指令

交互说明：

- 支持对单模块执行继电器开/关控制
- 控制后刷新设备详情与近期命令
- 指令反馈以“已创建、已下发、已执行、失败、排队中”为准

### 5.6 个人设置页

目标：

- 支持当前登录用户修改密码

接口：

- POST /api/v1/users/me/change-password

说明：

- 修改成功后提示重新登录
- 前端需做原密码、新密码、确认密码校验

## 六、权限与菜单策略

首期采用“双角色同时支持”的方案：

- super_admin：可访问首期全部核心页面
- manager：可访问首期全部核心页面，但数据范围由后端控制
- device_user：前端先兼容角色字段，但首期不单独设计差异化页面

首期菜单建议如下：

- 仪表盘
- 报警记录
- 设备管理
- 个人设置

说明：

- 首期不为 super_admin 单独增加“用户管理”“日志中心”“任务调度”等菜单
- 等第二阶段再扩展管理员专属后台能力

## 七、联调与交互约定

- 所有请求统一带 Bearer Token
- 401 统一登出并跳回登录页
- 接口错误统一采用全局错误提示 + 页面级兜底状态
- 仪表盘和设备详情页使用轮询刷新，默认 30 秒一次
- 首期不引入 WebSocket
- 图表仅消费后端已有统计数据，不在前端自行推导额外业务统计
- 分页列表以后端 pagination.total / limit / offset 为准

## 九、测试与验收标准

首期测试重点如下：

1. 认证链路
   - 未登录无法访问受保护页面
   - 正确账号密码可成功登录
   - token 失效后可自动退出
2. 仪表盘
   - 首页可展示统计信息、图表、近期报警、近期指令
   - 请求失败时页面有明确反馈
3. 报警记录
   - 支持分页
   - 筛选条件变化后可重新查询
   - 空数据状态展示正确
4. 设备管理
   - 可查看设备列表
   - 可进入设备详情
   - 模块状态展示正确
5. 手动控制
   - 可发起继电器开/关控制
   - 操作后有成功或失败反馈
   - 最近指令列表可反映控制结果
6. 个人设置
   - 修改密码校验正确
   - 修改成功后可重新登录

首期验收标准：

- 核心页面可正常打开
- 关键接口全部可联调
- 登录、查询、控制、修改密码四条主流程打通
- 页面具备基本后台可用性与错误兜底能力

## 十、默认假设

- 首期 Web 端直接从零开始搭建，不复用历史前端代码
- 后端现有 /api/v1 接口为首期联调基线
- 首期以桌面后台为主，不单独做移动端适配优化
- 当前 manager 角色视为普通业务账号
- 第二阶段再扩展用户管理、设备分组、日志中心、任务运维与管理员专属能力

## 十一、结论

当前项目已经具备进入 Web 端开发的条件。首期应围绕“登录、仪表盘、报警记录、设备详情、手动控制、修改密码”尽快完成前端工程与后端联调，优先形成一个可演示、可验证、可继续扩展的后台 MVP。后续再在此基础上逐步补齐管理员能力、运维能力与更复杂的系统管理功能。
