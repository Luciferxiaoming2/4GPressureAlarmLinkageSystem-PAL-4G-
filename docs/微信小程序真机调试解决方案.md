# 微信小程序真机调试解决方案

## 一、问题背景

本项目在微信开发者工具模拟器中可以正常联调，但在真机调试时，小程序长时间加载后提示：

- 网络请求失败，请检查网络连接

同时出现过以下现象：

- 小程序本地配置使用 `127.0.0.1:8001`
- 电脑和手机连接同一个热点
- 真机无法访问接口
- 手机浏览器也无法直接打开电脑上的本地服务地址

## 二、根因分析

这次问题的根因不是小程序页面代码本身，而是本地联调网络链路。

### 2.1 `127.0.0.1` 不能用于真机

小程序里如果配置：

```js
http://127.0.0.1:8001
```

那么在真机上，`127.0.0.1` 指向的是手机自己，不是开发电脑。

因此：

- 开发者工具模拟器可用
- 真机一定不可用

### 2.2 后端如果只监听 `127.0.0.1`，手机也无法访问

如果后端这样启动：

```powershell
python -m uvicorn app.main:app --host 127.0.0.1 --port 8001
```

那么只有电脑本机能访问，局域网设备不能访问。

真机联调时，后端至少要改成：

```powershell
python -m uvicorn app.main:app --host 0.0.0.0 --port 8001
```

### 2.3 即使监听正确，局域网仍然可能不通

这次已经确认：

- `netstat` 显示 `0.0.0.0:8001 LISTENING`
- 但手机浏览器仍然打不开 `http://电脑IP:8001`

这通常说明问题在：

1. Windows 防火墙拦截入站端口
2. 热点做了设备隔离
3. 手机和电脑虽然在同一热点下，但实际上不能相互访问

## 三、排查步骤

### 步骤 1：确认后端监听状态

```powershell
netstat -ano | findstr :8001
```

如果看到：

```text
0.0.0.0:8001 LISTENING
```

说明服务已经监听所有网卡。

### 步骤 2：确认手机浏览器能否访问本地服务

先找到电脑当前热点下的 IP，例如：

```text
10.150.141.4
```

然后在手机浏览器访问：

- `http://10.150.141.4:8001/docs`
- `http://10.150.141.4:8001/api/v1/health`

如果手机浏览器都打不开，问题就不在小程序代码，而在本地网络链路。

### 步骤 3：如果无管理员权限，不要继续死磕防火墙

本次尝试过添加 Windows 防火墙规则：

```powershell
New-NetFirewallRule -DisplayName "PAL_4G_8001" -Direction Inbound -Protocol TCP -LocalPort 8001 -Action Allow -Profile Private
```

但返回：

```text
Access is denied
```

说明当前终端没有管理员权限。

此时继续走局域网方案成本很高，应改走 HTTPS 穿透方案。

## 四、最终解决方案：使用 cloudflared 暴露本地后端

### 4.1 适用原因

`cloudflared` 可以把本地后端：

```text
http://127.0.0.1:8001
```

映射成一个可被手机和真机小程序访问的 HTTPS 域名，例如：

```text
https://commands-intimate-acne-rosa.trycloudflare.com
```

优点：

- 不依赖局域网互通
- 不依赖热点是否做设备隔离
- 不要求先解决 Windows 防火墙权限
- 微信小程序真机对 HTTPS 域名的兼容通常比本地 IP 更稳

### 4.2 启动方式

先启动后端：

```powershell
cd D:\project\code\PAL_4G\backend
D:\uv\venvs\pal_4g\Scripts\python.exe -m uvicorn app.main:app --host 0.0.0.0 --port 8001
```

再启动 cloudflared：

```powershell
cd D:\Tools\cloudflare
.\cloudflared-windows-amd64.exe tunnel --url http://127.0.0.1:8001
```

输出中会给一个临时 HTTPS 域名。

### 4.3 小程序配置修改

在小程序配置中改成：

```js
const API_ORIGIN = 'https://commands-intimate-acne-rosa.trycloudflare.com'

export const APP_CONFIG = {
  API_BASE: `${API_ORIGIN}/api/v1`,
  WS_URL: `${API_ORIGIN.replace('https://', 'wss://')}/api/v1/ws/events`,
}
```

### 4.4 注意事项

1. 后端终端不能关
2. `cloudflared` 终端不能关
3. 临时 Tunnel 地址不是固定的，重新启动后可能变化
4. 每次地址变化后，都需要重新改小程序配置并重新编译

## 五、这次得到的经验

### 5.1 真机调试不要默认使用 `127.0.0.1`

这只适合开发者工具本机模拟，不适合真机。

### 5.2 真机网络异常先用手机浏览器验证

如果手机浏览器都访问不了电脑服务：

- 小程序一定也访问不了
- 此时不要先怀疑接口业务代码

### 5.3 局域网调试不稳定时，优先用 HTTPS 隧道

当出现以下情况时，优先考虑 `cloudflared`、`ngrok`、`frp`：

- 热点环境
- 没有管理员权限改防火墙
- 局域网设备隔离
- 微信真机对本地 IP 请求不稳定

## 六、推荐后续优化

1. 把小程序接口地址改成环境变量或本地配置文件，不要硬编码在源码中
2. 增加一个开发环境说明文档，明确：
   - 模拟器配置
   - 真机配置
   - cloudflared 使用方式
3. 后续如需长期真机联调，可考虑固定 HTTPS 测试域名，而不是每次依赖临时 Tunnel

## 七、结论

本次微信小程序真机调试失败，根因是本地网络链路而不是业务代码。

最终采取的稳定方案是：

1. 后端监听 `0.0.0.0:8001`
2. 使用 `cloudflared` 把本地后端暴露为 HTTPS 域名
3. 小程序接口地址改为该 HTTPS 域名后重新编译

这套方案更适合作为后续本项目真机联调的默认参考路径。
