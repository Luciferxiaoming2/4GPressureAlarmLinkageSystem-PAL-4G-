- 项目名称：4G 压力报警联动控制系统

## 二、后端目标范围

根据现有需求文档，后端需要承担以下核心职责：

1. 提供 Web 管理后台与微信小程序共用的业务 API。
2. 维护设备与云端的长连接通信，接收设备状态、报警事件，并下发控制指令。
3. 实现报警联动引擎，在单模块报警时联动同组其他模块执行继电器动作。
4. 存储设备状态、报警记录、操作日志、用户与设备绑定关系。
5. 提供账号体系、角色权限、登录鉴权能力。
6. 提供后台运转监控、历史记录查询、设备管理、远程控制等服务能力。

## 三、技术栈约定

依据项目文档，后端默认技术栈如下：

- 框架：FastAPI
- ASGI：Uvicorn
- ORM：SQLAlchemy 2.x
- 数据库：SQLite + aiosqlite
- 鉴权：python-jose + passlib
- 设备通信：paho-mqtt
- 定时任务：APScheduler
- 数据校验：Pydantic v2
- 日志：Python logging

说明：

- 当前项目文档明确推荐 SQLite，适合前期快速交付与本地验证。
- 若后续设备量、并发量、日志量明显上升，建议再评估迁移到 MySQL 或 PostgreSQL。

## 八、待确认与风险项

### 8.1 通信协议细节未落地

现状：

- 需求文档提到设备侧“已有协议”，但仓库中没有看到协议字段、Topic 规范、报文格式、鉴权方式、应答机制等细节。

影响：

- 无法准确完成 MQTT 消息解析与设备指令下发。
- 可能导致后端数据模型与实际报文不匹配。

建议：

- 尽快补充协议文档、报文样例、主题设计、上下行字段定义。

### 8.2 报警恢复与手动控制冲突规则需要细化

现状：

- 文档已明确“报警解除自动恢复”，但手动控制与自动联动冲突处理逻辑还不够细。

影响：

- 继电器最终状态判定容易产生歧义。

建议：

- 明确优先级规则，例如：
  - 报警联动优先于手动关闭
  - 手动控制是否允许覆盖自动联动
  - 报警未解除时是否允许人工解除继电器

### 8.3 数据库选型适合前期，不一定适合中后期

现状：

- 当前文档默认 SQLite。

影响：

- 前期开发快，但设备量上来后，写入压力、并发查询、日志量可能成为瓶颈。

建议：

- 第一阶段可按 SQLite 落地。
- 第二阶段根据实际接入规模评估迁移方案。

### 8.4 微信授权登录方案未细化

现状：

- 需求文档提到支持微信授权登录绑定已有设备账号，但未描述绑定流程。

影响：

- 后端接口设计与账号体系可能需要预留额外字段与授权绑定表。

建议：

- 后端第一阶段先实现账号密码登录。
- 微信授权绑定在第二阶段补充设计与实现。
