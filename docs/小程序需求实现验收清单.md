## 五、当前缺口

当前主要缺口不在“有没有代码”，而在“有没有完成真机验收和自动化测试”：

1. 小程序没有独立自动化测试工程。
2. 订阅消息依赖真实模板 ID、字段映射和小程序后台配置。
3. 微信登录依赖真实 `AppID/AppSecret` 与真机环境。
4. 当前 `miniprogram/utils/config.js` 使用本地 `127.0.0.1` 地址，只适合本机开发，不适合真机外网联调。

## 六、测试建议

### 6.1 当前最适合的测试方式

1. 后端接口测试：继续使用 `pytest`
2. 小程序功能联调：微信开发者工具
3. 真机能力测试：微信扫码真机预览/真机调试

### 6.2 为什么小程序不直接用 pytest

`pytest` 适合后端接口、服务逻辑、任务调度和数据库行为验证；不适合直接覆盖小程序运行时能力，比如：

- `uni.login`
- `uni.requestSubscribeMessage`
- `uni.scanCode`
- `uni.connectSocket`
- 页面跳转、页面生命周期、真机弱网和授权弹窗

### 6.3 如果要补小程序自动化

优先考虑：

- `miniprogram-automator`
- `uni-automator`

不建议直接把 Web 的 `Playwright` 当成小程序自动化主方案。

## 七、建议验收顺序

1. 启动后端并确认 Swagger、登录、首页摘要、设备接口可用
2. 在微信开发者工具中运行小程序，先走账号密码登录主链路
3. 验证首页、设备列表、设备详情、多模块状态、继电器控制
4. 验证报警筛选、分页、下拉刷新、加载更多
5. 验证绑定设备、移除设备、修改密码、退出登录
6. 在真机环境验证微信登录、微信绑定、订阅授权、扫码填充
7. 用真实或模拟报警触发一次订阅消息派发闭环

## 八、结论

当前小程序已经满足“首期主流程已实现并已接入后端接口”的判断，可以进入以微信开发者工具和真机为主的小程序专项联调阶段。

如果要继续提高交付确定性，下一优先级应是：

1. 真机联调与环境配置收口
2. 补小程序自动化测试
3. 修正所有过时文档，避免后续协作误判
