# PAL_4G 部署文档

本文档基于当前仓库代码、`deploy/` 目录模板以及真实设备接入路线整理，用于说明项目上线时需要准备什么、替换什么、当前还缺什么。

## 1. 三端部署方式对照

| 端 | 部署方式 | 是否需要打包 | 最终上线形态 |
| --- | --- | --- | --- |
| Web 前端 | 本地或 CI 构建后，把静态文件放到 Nginx 目录 | 需要 | `dist` 静态文件 |
| 后端 | 部署源码、安装依赖、填写 `.env`，再用 `uvicorn + systemd` 运行 | 不需要前端那种打包 | 持续运行的 Python 服务 |
| 微信小程序 | 编译成微信小程序产物后，通过微信开发者工具上传、提审、发布 | 需要编译 | 微信平台上的小程序版本 |

补充说明：

- Web 是典型的“先打包，再部署产物”。
- 后端不是打包成静态文件，而是“部署代码并运行服务”。
- 小程序也不是像后端那样在服务器上直接运行源码，而是编译后发布到微信平台。

## 2. 端口清单

| 服务 | 默认端口 | 是否必须公网开放 | 说明 |
| --- | --- | --- | --- |
| Web 前端 | `5173` | 否 | 本地开发端口，正式环境通常走 `80/443` 反向代理 |
| 后端 API | `8001` | 否 | 本地开发端口，正式环境通常走 `80/443` 反向代理到后端 |
| MQTT Broker 明文 | `1883` | 可选 | 仅联调用，正式环境不建议直接暴露 |
| MQTT Broker TLS | `8883` | 是 | 真实设备直连建议正式使用该端口 |
| HTTPS | `443` | 是 | Web、API、证书校验、管理入口 |
| HTTP | `80` | 可选 | 可仅用于跳转到 `443` |

说明：

- 正式环境不建议让用户直接访问 `5173` 或 `8001`
- 正式环境建议统一通过域名和 `443`
- 真实设备 MQTT 直连优先使用 `8883`

## 5. 部署人员必须替换的内容

### 5.1 后端配置

以下项必须改成真实值，不能直接沿用模板：

| 配置项 | 为什么必须改 |
| --- | --- |
| `SECRET_KEY` | 默认值不安全 |
| `DEFAULT_ADMIN_PASSWORD` | 默认管理员密码不能直接上线 |
| `MQTT_BROKER_HOST` | 当前模板只是推荐域名 |
| `MQTT_BROKER_PORT` | 要和最终 Broker 实际端口一致 |
| `MQTT_USERNAME` | 当前是占位值 |
| `MQTT_PASSWORD` | 当前是占位值 |
| `MQTT_TLS_CA_CERTS` | 要替换成服务器真实证书路径 |
| `MQTT_TLS_SERVER_HOSTNAME` | 要和 Broker 真实域名一致 |
| `WECHAT_APP_ID` | 当前是占位值 |
| `WECHAT_APP_SECRET` | 当前是占位值 |
| `WECHAT_SUBSCRIBE_TEMPLATE_ID` | 当前是占位值 |

### 5.2 Web 配置

| 配置项 | 为什么必须确认 |
| --- | --- |
| `VITE_API_BASE` | 生产环境建议保留 `/api/v1`，如果网关路径不同要同步调整 |

### 5.3 Nginx 配置

以下内容当前都只是模板值：

| 配置项 | 当前值 | 部署时要改什么 |
| --- | --- | --- |
| `server_name` | `usr-iot.endpage.net` | 改成真实站点域名 |
| `ssl_certificate` | Let’s Encrypt 示例路径 | 改成真实证书路径 |
| `ssl_certificate_key` | Let’s Encrypt 示例路径 | 改成真实私钥路径 |
| `root` | `/var/www/pal4g/web/dist` | 改成实际 Web 静态目录 |

### 5.4 systemd 配置

以下内容必须按服务器环境调整：

| 配置项 | 当前值 | 部署时要改什么 |
| --- | --- | --- |
| `User` / `Group` | `www-data` | 改成实际运行用户 |
| `WorkingDirectory` | `/opt/pal4g/backend` | 改成项目真实路径 |
| `EnvironmentFile` | `/opt/pal4g/backend/.env` | 改成后端 `.env` 真实路径 |
| `ExecStart` | `/opt/pal4g/.venv/bin/python ...` | 改成服务器 Python/venv 真实路径 |

## 7. 真实设备联调还需要客户或现场提供的内容

| 类别 | 是否必须 | 需要提供的内容 | 说明 |
| --- | --- | --- | --- |
| 真实设备 MQTT 参数修改权限 | 必须 | 能修改设备 MQTT 配置 | 否则设备无法切到自有 Broker |
| 真实设备 SN | 必须 | 每台设备的 SN | 后端建档和 topic 识别依赖它 |
| 真实设备 IMEI | 建议 | 每台设备的 IMEI | 便于排查和辅助识别 |
| 状态上报样例 | 必须 | 一条真实 `status` payload | 用于核对字段映射 |
| 报警上报样例 | 必须 | 一条真实 `alarm` payload | 用于核对报警字段和联动触发 |
| 执行反馈样例 | 必须 | 一条真实 `feedback` payload | 用于核对命令回执处理 |
| 断开命令格式 | 必须 | 真实设备如何表示“断开” | 如果不是当前 JSON，后端需调整 |
| 恢复命令格式 | 必须 | 真实设备如何表示“恢复” | 如果不是当前 JSON，后端需调整 |
| TLS 兼容性 | 必须确认 | 真实设备是否支持校验证书 | 决定最终 Broker TLS 配置 |
| 现场网络 | 必须确认 | SIM 卡能否访问公网 Broker | 不通则无法联调 |

## 8. 推荐部署顺序

1. 准备 Linux 服务器、域名、DNS 和证书
2. 先部署 MQTT Broker，推荐从 `deploy/emqx-docker-compose.yml` 起步
3. 按 `deploy/backend.env.production.example` 填好后端生产配置
4. 安装后端依赖并用 `deploy/pal4g-backend.service` 托管
5. 构建 Web 并把静态文件部署到 Nginx 站点目录
6. 按 `deploy/nginx.pal4g.conf` 调整域名、证书和静态目录
7. 放开 `443`、`8883`，必要时临时放开 `1883`
8. 让真实设备改连自有 Broker，并按 `pal4g/devices/{device_sn}/...` 主题规范联调
9. 最后做 Web、API、MQTT 和真机验收

## 9. 一句话结论

当前后端代码已经基本准备好。  
真正阻塞正式部署和真机联调的，主要不是后端逻辑，而是：

- 服务器未部署
- MQTT Broker 未部署
- TLS 证书未准备
- 域名和 DNS 未配置
- 微信正式配置未补齐
- 真实设备未切换到自有 Broker
- 真实报文和控制命令格式还没最终确认
